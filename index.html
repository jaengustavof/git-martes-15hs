Areas de GIT

Working Directory

Area de trabajo hace referencia a la carpeta donde creamos el repositorio. Muy rar vez utilizaremos el area de comandos en esta zona, pero es donde ralizaremostodas las modificaciones relacionadas con nuestro proyecto

  • dir || ls muestra el contenido del directorio actual
  • mkdir <dir> Crea un directorio con el nombre especificado
  • cd <dir> Cambia el directorio actual
  • git init Inicializa un repositorio de GIT, creando una carpeta oculta que lleva el nombre .git
  • git status Verifica si hay modificaciones en el Working Directory
  • rm <dir> Elimina un archivo. Pero podmos especificar con parametros que elimine una carpeta, por ejemplo: rm -rf .git

Staging Area (INDEX)

El area de control de cambios corresponde al archivo que se encuentra en la carpeta .git que se crea al iniciar un repositorio. Aqui se registran los archivos a los que vamos a hacerle seguimiento y es el punto de comparacion para confirmar dichis cambios. Es necesario agregar los archivos con los que estemos trabajando a dicha area para avanzar con el historial de cambios, ya que git al utilizar el comandos git addrealiza una captura de pantalla.

  • git status

    compara el Working Directory con el contenido del Staging Area (index) y mustra en pantalla el estado de los archovos del Working Directory.

    • untraked

      El archivo no se encuentra en el index. Por lo tanto no se le está haciendo seguimiento.

    • tracked

      Los archivos / cambios fueron agregados al Index y están listos para ser confirmados.

    • modified

      El archivo que se encuentre en el Working Directory difiere con los cambios almacenados en el Index.

    • commited

      Los cambios fuern confirmados y almacenados en el repositorio.

  • git add <file>

    Agrega en el index los cambios/archivos especificados, para agregar todo el contenido de un directorio podemos utilizar el comando git add .

  • git restore <file>

    Restblece los cambios en el working directory, a la ultima version confirmada del archivo.

  • git reset <file> --cached

    Quita del index el archivo seleccionado, pero mantiene la copia en el working directory.

Repository (Local/Remote)

El repositorio corresponde a la base de datod donde se almacenan los cambios en formato BLOB(Binary Object File) y es el lugar al que podemos acceder cada vez que necesitemos revisar un estado anterior de nuestro proyecto. Esto es posible gracias de los commits identificados mediante un numero de HASH.

  • git commit

    Abre el editor de codigo predeterminado para una confirmacion de cambios. Podemos utilizar comandos como git commit -am "msg" para agregar al index los cambios de los archivos con seguimiento al index y generar un mensaje de commit.

  • git diff <commit>

    compara el working directory con la referencia especificada, esta puede ser relativa HEAD^(ansestro) o HEAD~2(anteriores) o absoluta utiliazndo el HASH

  • git blame <file>

    Muestra el archivos seleccionao con el nombre de usuario, fecha y hora en las que se realizaron modificaciones en el mismo. En caso de haber cambios sin confirmacion muestra la leyenda Not Comit Yet

  • git checkout <commit>

    Permite que naveguemos en el historial de cambios para ver las versiones de nuestro proyecto.

Apuntadores

HEAD

Apuntador movil que nos indica donde nos encontramos dentro del historial de commits. Se utiliza como referencia y tambien para movernos dentro del proyecto, así de esta manera controlamos todos las modificaciones que fue sufriendo el mismo. Un comando que utiliza esta referencia es git checkout por ejemplo: git checkout git HEAD~2 para ir dos cambios atras en el historial.

BRANCH

Las ramas son apuntadores dinamios que siempre se posicionan en el ultimo commit de su historia. Las mismas deben compartir un commit en comin para tener una referencia de posicionamiento. Las mismas se utilizan para implementar modificaciones experimentales de desarrollo, prueba o simplemente para revisar un problema y una vez solucionado el mismo, integrarlo a la rama principal.
